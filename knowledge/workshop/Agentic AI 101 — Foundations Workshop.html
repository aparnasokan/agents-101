Clearing the Fog

🔎
Sources

Five terms everyone in this room should mean the same thing when they say. Click each card to reveal.

🧠

LLM

tap to reveal
A statistical engine trained on text that predicts the most useful next sequence of words. No memory. No goals. No awareness — only pattern recognition at massive scale.
✗ NOT
A search engine, database, reasoning machine, or brain.
🤖

AI Agent

tap to reveal
An AI system that perceives its environment, makes decisions, and takes actions to achieve a goal — autonomously, over multiple steps, without a human approving each move.
✗ NOT
A chatbot, a single API call, or a workflow trigger.
⚡

Agentic AI

tap to reveal
AI systems designed for autonomous, goal-directed behavior across complex, multi-step tasks. The system can plan, act, observe results, and adapt — without step-by-step human guidance.
✗ NOT
A single prompt-response exchange, no matter how sophisticated the prompt.
🔄

Agentic Workflow

tap to reveal
A sequence of AI-driven steps where outputs from one action feed into the next — enabling complex task completion across tools, data sources, and decision points.
✗ NOT
An RPA script or hardcoded process. Agentic workflows adapt; scripts only execute.
🕸

Multi-Agent System

tap to reveal
Multiple specialized AI agents collaborating — each with defined roles — coordinated by an orchestrator to solve tasks no single agent could handle alone.
✗ NOT
One model with multiple prompts. Separate agents have separate contexts and capabilities.

The Maturity Curve

Where organizations typically go on this journey

Wave 1
Agentic Applications
One agent, one job. It has tools, memory, and a goal — and it figures out the steps itself.
A support agent that reads your knowledge base and drafts a response
A code assistant that writes a function, runs the tests, and fixes what breaks
A scheduling bot that checks three calendars and books the meeting
This is where most companies actually are today.
Wave 2
Agentic Processes
Multiple agents own an entire workflow end-to-end. A human kicks it off; agents handle the rest.
Customer calls in → agent triages, pulls data from 3 systems, drafts resolution, checks compliance, sends
New hire onboarding: collect docs, provision accounts, schedule orientation — no handoffs to humans
Insurance claim: intake, validate, adjudicate, pay — straight-through for simple claims
A few leaders are here. Most who claim to be aren't.
Wave 3
Agentic Enterprise
Agents across the org share context, learn from each other's outcomes, and self-optimize without being told to.
A billing agent detects an anomaly and flags the retention agent, which reaches out before the customer calls
The system notices a pattern across thousands of interactions and adjusts its own routing rules
Agents negotiate priorities across departments — supply chain agents coordinate with sales agents in real time
Nobody's here yet. This is the north star.

What Is an Agent Made Of?

Four building blocks every agent needs. The difference between a demo and a production agent is how well you engineer each one.

🧠
Model
The reasoning engine. The LLM that interprets context, plans steps, and generates outputs.
GPT-4, Claude, Gemini, or a fine-tuned domain model
🔧
Tools
What the agent can do. APIs, databases, search, code execution — the hands that act on the world.
Check eligibility API, query knowledge base, send notification
📈
State & Memory
What the agent knows. Working context, session history, long-term memory, and structured artifacts.
Customer profile, conversation history, previous recommendations
🛡
Governance
What the agent is allowed to do. Guardrails, policies, human-in-the-loop gates, and compliance rules.
PII masking, escalation thresholds, approval workflows
After the anatomy cards, make this point: "Everyone focuses on the model. But the model is maybe 10% of what makes an agent work. The real engineering is in the tools, the state management, and the governance. That's what separates a demo from production."

Context Engineering — The Hidden Discipline

Context is not just "what you put in the prompt." It's a compiled view over tiered state — and the quality of your agent's context determines the quality of its output.

The Four Context Tiers
Each tier has different scope, lifetime, and engineering requirements.
T1
Working Context
The current task. What the agent is doing right now — the active goal, intermediate results, and scratchpad.
T2
Session Context
This conversation. Everything that's happened in the current interaction — messages, decisions, tool results.
T3
Memory
Cross-session knowledge. Customer preferences, past resolutions, learned patterns — what persists between conversations.
T4
Artifacts
Structured outputs. Documents, data objects, and reusable assets the agent has created or consumed.
This is a concept from Google's agent research. The key insight: "Most teams only think about Tier 1 and 2. But agents that scale need Tier 3 and 4 — persistent memory and reusable artifacts. That's where long-horizon agents get their power."

If pressed for time, skip the visual and just land the headline: "Context engineering is the real discipline behind production agents."
Walk through each card — have the group guess before you flip. The analogy that lands consistently: "a brilliant intern with amnesia." Incredibly capable on any single task, but needs full context every time, has no persistent goals, and won't act without being asked. That's an LLM. When you give it a goal, tools, and the ability to act — that's an agent.

For the maturity curve: anchor your audience with "Most organizations are beginning Wave 1. The vocabulary we're building today is what makes Wave 2 and 3 possible."

The 5 Design Patterns

🔎
Sources
Anthropic: "Building Effective Agents" — composable patternsOpenAI: "A Practical Guide to Building Agents" — orchestration, guardrails

How agentic systems are actually architected. These five patterns cover the vast majority of real-world agentic designs. Click any card to expand the full breakdown.

⚠️
Principle #1: Start with the simplest pattern that works.
Use a single LLM call if it solves the problem. Add patterns only when you hit a real limitation — not because the architecture diagram looks more impressive. Every lab (Anthropic, Google, OpenAI) converges on this: complexity is a cost, not a feature.
PATTERN 01
Prompt Chaining
Sequential steps, each feeding the next
HOW IT WORKS
A complex task is decomposed into a sequence of focused LLM calls. Each call processes the output from the previous step, creating a pipeline where every stage does one thing well. Crucially, you can insert quality gates between steps — programmatic checks that verify the output before it moves on. If a step fails a gate, the chain can stop early, retry, or reroute.
KEY CHARACTERISTICS
Error isolation — When something fails, you know exactly which step broke. Each step can be debugged and improved independently.
Quality gates — Insert validation checkpoints between steps: format checks, classification confidence thresholds, compliance rules.
Focused context — Each LLM call gets a smaller, focused prompt. Simpler tasks = higher accuracy per step.
Latency tradeoff — Total time is the sum of all steps. You're trading speed for accuracy and control.
WHEN TO USE vs. WHEN NOT TO
USE WHEN
  • Steps have sequential dependencies (B needs A's output)
  • Each stage benefits from focused, specialized attention
  • Quality gates between stages are valuable
  • You need auditability — tracing exactly where a decision was made
DON'T USE WHEN
  • Steps are independent of each other (use Parallelization)
  • You need real-time speed — sequential adds up
  • The task is simple enough for a single well-crafted prompt
EXAMPLES
Intent classification → context enrichment → response generation Document ingestion → entity extraction → summarization Customer message → sentiment analysis → priority scoring → routing
ANALOGY
Assembly line — each station does one thing well before passing the piece to the next. A quality inspector can sit between any two stations.
Common mistake: Chaining steps that don't actually depend on each other. If Step 2 doesn't need Step 1's output, you're adding unnecessary latency. Use Parallelization instead.
PATTERN 02
Routing
Classify, then direct to the right specialist
HOW IT WORKS
An initial classification step analyzes the input and routes it to the most appropriate specialized handler. Each specialist is optimized for its domain — different prompts, different tools, different models if needed. The classifier makes a one-time, stateless decision. Once it hands off, its job is done. The specialist handles the rest independently.
KEY CHARACTERISTICS
Separation of concerns — Each specialist is optimized for one domain. Better prompts, better tools, better results.
Classification is the bottleneck — The quality of the entire system depends on accurate routing. Invest heavily here.
Stateless handoff — The classifier doesn't manage what happens after routing. This is what distinguishes it from orchestration.
Multiple strategies — Classification can be LLM-based, rule-based, ML model, or hybrid. Pick what fits your accuracy needs.
WHEN TO USE vs. WHEN NOT TO
USE WHEN
  • Inputs have clearly distinct types requiring different handling
  • Specialization outperforms a generalist approach
  • You want to add new specialists without touching others
  • Different domains need different tools, models, or guardrails
DON'T USE WHEN
  • One generalist prompt handles all cases well enough
  • Classification boundaries are fuzzy or constantly shifting
  • You need ongoing coordination after the handoff (use Orchestrator)
EXAMPLES
Detect intent (billing / technical / retention) → route to specialist agent Classify document type (invoice / contract / support ticket) → process accordingly Detect language → route to language-specific model and prompt
ANALOGY
Triage nurse — quickly assesses what kind of help you need, then sends you to the right specialist. The nurse doesn't do the surgery.
Common mistake: Over-specializing. If you have 15 specialists and the classifier can't reliably distinguish between them, you'll get worse results than a single generalist. Start with 3-5 clear categories.
PATTERN 03
Parallelization
Run tasks simultaneously, then aggregate
HOW IT WORKS
Multiple LLM calls fire at the same time. An aggregation step combines results into one coherent output. This comes in two flavors:
Sectioning — The task is split into independent subtasks. Each runs in parallel. Results are merged. Example: Pull account data, device data, and promo data simultaneously.
Voting — The same task runs multiple times (same or different approaches). Results are compared, and the best or most common is selected. Example: Three models evaluate a response; majority wins.
KEY CHARACTERISTICS
Speed — Total latency = slowest task, not sum of all tasks. Critical for real-time applications.
Independence required — Tasks cannot depend on each other. If Task 2 needs Task 1's output, use Chaining instead.
Aggregation strategy matters — Merging results is non-trivial. Define how conflicts are resolved before you build.
Higher cost — You're making N calls instead of 1. The speed benefit must justify the compute cost.
WHEN TO USE vs. WHEN NOT TO
USE WHEN
  • Multiple independent data sources need to be gathered
  • Speed is critical and tasks don't depend on each other
  • You want consensus from multiple models or approaches (voting)
  • The aggregation logic is clear and deterministic
DON'T USE WHEN
  • Tasks have sequential dependencies
  • The aggregation strategy is unclear or subjective
  • Cost is a hard constraint and speed isn't
EXAMPLES
Simultaneously pull account status + device eligibility + active promotions Run compliance check + sentiment analysis + tone review in parallel Three models score a response; take the majority verdict (voting)
ANALOGY
Research team — everyone works their section in parallel; a lead synthesizes the findings into one report.
Common mistake: Assuming tasks are independent when they aren't. If the promo data depends on knowing the account tier, that's a hidden dependency. Map your data flows first.
PATTERN 04
Orchestrator–Subagents
A coordinator delegates to specialists and manages dependencies
HOW IT WORKS
A central orchestrator LLM takes a complex goal and dynamically decomposes it into subtasks. It assigns each to a specialist sub-agent, manages dependencies between them (some may run in sequence, others in parallel), waits for results, and synthesizes the final output. The key difference from other patterns: the orchestrator reasons about what needs to happen — the task breakdown isn't predetermined.
KEY CHARACTERISTICS
Dynamic decomposition — Unlike Chaining (fixed sequence) or Parallelization (fixed split), the orchestrator decides the plan based on the specific input.
Dependency management — The orchestrator understands which subtasks depend on others and sequences them accordingly.
Specialist leverage — Sub-agents can be deeply specialized: different models, tools, knowledge bases, and guardrails per agent.
Composes other patterns — An orchestrator might use Routing to pick agents, Parallelization to run them, and Chaining for sequences. It's the "meta-pattern."
WHEN TO USE vs. WHEN NOT TO
USE WHEN
  • The task is complex with interdependent subtasks
  • The plan can't be known in advance — it depends on the input
  • Specialist agents outperform a single generalist
  • You need intelligent coordination, not just parallel execution
DON'T USE WHEN
  • The workflow is simple and predictable (use Chaining)
  • Tasks are independent with no coordination needed (use Parallelization)
  • The overhead of an orchestrator outweighs the task complexity
EXAMPLES
Personalization agent: journey analysis → offer qualification → sentiment gate → assemble Customer resolution: diagnose issue → check policies → generate solution → confirm Report generation: gather data → analyze trends → draft narrative → format
ANALOGY
Project manager — understands the full picture, decides who does what and in what order, resolves blockers, and assembles the final deliverable.
Common mistake: Using an orchestrator when a simple chain would do. Every orchestrator call adds latency and cost. If the sequence of steps is always the same regardless of input, use Prompt Chaining — it's simpler, faster, and easier to debug.
PATTERN 05
Evaluator–Optimizer
Generate, evaluate, refine — loop until quality is met
HOW IT WORKS
A generator produces output. A separate evaluator scores it against defined criteria. If it fails, structured feedback is sent back for revision. The loop continues until the output passes all quality gates — or hits a maximum iteration count. The key innovation: the evaluator provides specific, actionable feedback, not just a pass/fail. This turns the loop into genuine refinement, not random retries.
KEY CHARACTERISTICS
Measurable criteria — The evaluator needs clear, testable rules. Vague criteria like "make it better" create infinite loops.
Structured feedback — The evaluator tells the generator exactly what failed and why. This is what makes revision productive.
Iteration limits — Always set a max iteration count. Without it, you risk infinite loops and runaway costs.
Converging quality — Each loop should measurably improve the output. If quality isn't converging, the criteria or feedback are broken.
WHEN TO USE vs. WHEN NOT TO
USE WHEN
  • Quality must meet specific, measurable criteria before use
  • Iterative refinement produces meaningfully better results than a single pass
  • You have clear evaluation rubrics (compliance rules, format specs, test suites)
  • The cost of bad output is higher than the cost of extra iterations
DON'T USE WHEN
  • Evaluation criteria are subjective or undefined
  • A single well-prompted pass produces good-enough quality
  • Latency is critical — loops add unpredictable time
  • The generator can't meaningfully improve from feedback
EXAMPLES
Draft proactive message → check compliance + brand rules → revise until pass Generate code → run test suite → fix failures → re-test until green Create customer summary → verify accuracy against source data → correct errors
ANALOGY
Writer and editor — the writer drafts, the editor gives specific notes ("paragraph 3 contradicts the data"), the writer revises. This continues until the editor signs off.
Common mistake: Evaluating with vague criteria. If the evaluator says "this could be better" without specifying what, the generator will make random changes. Define exactly what "pass" looks like before building the loop.

Pattern Decision Guide

If your situation is... Use this pattern
Steps must happen in sequence — each depending on the previous outputPrompt Chaining
Different request types need fundamentally different handlingRouting
Multiple independent tasks need to complete and their results combinedParallelization
Complex task with interdependent subtasks requiring intelligent coordinationOrchestrator–Subagents
Output must meet specific quality criteria before it's considered doneEvaluator–Optimizer

Patterns Are Composable

Real systems rarely use a single pattern. They combine them. Here are three common compositions:

Routing + Prompt Chaining
Classify the input, then run a specialized chain for that type. Each route has its own multi-step pipeline.
Classify intent → Route to specialist → Chain: enrich → reason → respond
Orchestrator + Parallelization + Evaluator
The orchestrator gathers data in parallel, synthesizes a response, then runs it through a quality loop before delivering.
Orchestrator plans → Parallel data fetch → Synthesize → Evaluate ↺ → Deliver
Parallelization (Voting) + Evaluator
Generate multiple candidate responses in parallel. Evaluate each. Pick the best. If none pass, regenerate.
3 generators (parallel) → Evaluate all → Select best / retry
This is the core of the session. Walk through each card — expand one or two as demonstrations. For each pattern, anchor on the analogy and one concrete example. Spend no more than 2–3 min per pattern.

Key insight to land: "These patterns are composable. Real systems combine them — a routing pattern can direct to an orchestrator, which uses parallelization internally. You're learning letters, not full sentences. The sentences come next."

The "Two Flavors" section in Parallelization is a good discussion starter — ask: "Which flavor would you use for gathering customer data? What about checking response quality?" (Answer: Sectioning for data, Voting for quality.)

The "When NOT to use" sections are deliberately provocative — use them to show that choosing the wrong pattern is worse than no pattern at all.

How Agents Talk to Each Other

Four emerging protocols that standardize how agents connect. Click to expand.

MCP
Model Context Protocol
How agents use tools and access data. Agent-to-tool / agent-to-data.
What it standardizes: A universal interface for connecting an agent to external tools, databases, and APIs — so you don't have to write custom integrations for every tool. Uses JSON-RPC with capability negotiation.

Example: An agent discovers it can query a customer database, check inventory, and send notifications — all through the same protocol, regardless of who built each tool.
▶
A2A
Agent2Agent Protocol
How agents find and collaborate with each other. Agent-to-agent.
What it standardizes: Agent discovery (via agent cards that describe capabilities), task lifecycle management, and streaming results between agents. Enables agents built by different teams to collaborate without pre-integration.

Example: A billing agent publishes its capabilities. A customer service orchestrator discovers it, delegates a refund task, and receives status updates — all through a standard protocol.
▶
ACP
Agent Communication Protocol
An open, HTTP-based protocol for structured, multimodal messages, task delegation, context management, and observability across heterogeneous agents.
What it standardizes: Structured, multimodal messaging between agents over HTTP — covering task delegation, context passing, and built-in observability. Designed to work across heterogeneous agent frameworks without requiring a shared runtime.

Example: A planning agent delegates a research subtask to a specialist agent built on a completely different framework, passing rich context (text, images, structured data) and receiving streaming progress updates — all over standard HTTP endpoints.
▶
AP2
Agent Payments Protocol
When agents need verifiable authority and accountability. The advanced case.
What it standardizes: Authentication, authorization, and audit trails for agent-to-agent transactions that require verifiable accountability — beyond simple task delegation.

Example: An agent autonomously negotiates a vendor contract within pre-approved parameters, with every step cryptographically signed and auditable.
▶
Frame this as: "MCP is about tools. A2A is about teamwork. ACP is about interoperability. AP2 is about trust. Most teams start with MCP (connecting agents to tools), then add A2A or ACP for agent-to-agent communication, and AP2 only when agents need to make consequential decisions autonomously."

AP2 is still early-stage and niche — don't spend more than 30 seconds on it. MCP, A2A, and ACP are the ones that matter for most teams today. A2A and ACP overlap in the agent-to-agent space — A2A focuses on discovery and task lifecycle, while ACP focuses on structured messaging and multimodal context passing.

From Demo to Production

Designing an agent is one thing. Operating it is another. Three capabilities separate a demo from a production system.

📌
Versioning
Agents need version control like code. You need to know which version of your agent produced which output — and roll back when something breaks.
📊
Eval Loops
Systematic evaluation of agent outputs, not spot-checking. Automated test suites for agent behavior that run on every change, every deployment.
🔍
Trace Grading
Every agent action should be traceable and gradeable. When something goes wrong, you can replay the full decision chain and diagnose exactly where it failed.
This is the OpenAI perspective on operational maturity. Key line: "Everyone can build an agent demo in a day. The question is: can you run it in production for 6 months, know when it's drifting, and fix it without breaking everything else? That's what versioning, eval loops, and trace grading give you."

Don't go deep here — just plant the seed. This is the "what comes next" message.
Keep this section light — you're showing how the patterns connect to a real architecture, not presenting a technical spec.

Key line to land verbatim: "The patterns we just practiced in the quiz are the building blocks for each of these layers — and that's the work ahead."

If questions arise about specific technology choices, team ownership, or timelines: "Great question — that's exactly the detailed design work. This session gives us the shared vocabulary to have that conversation."
